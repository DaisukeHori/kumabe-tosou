## 0. 前提とスコープ

- モックアップは **静的HTML+インラインstyle**(dc-runtime形式)。本移行は「コピペ移植」ではなく **Tailwind v4(CSS-first)+ shadcn(base-nova)への翻訳作業**である。
- 現行adminは全ページ Server Component + facade直呼び出し(`dynamic="force-dynamic"`)。**この移行はプレゼンテーション層のみを触り、モジュール境界(facade)・`Result<T>`・エラーコード帯・既存テストを壊さない**ことを設計原則とする。
- モックアップには **現行に存在しない機能**(折り返し発信、メール返信、録音ファースト等)が混在している。本設計では「①見た目/IAの移行」と「②新機能」を明示的に分離し、②はリデザインIssueに**含めない**(別トラック起票候補として付録に列挙)。なお、問い合わせ→顧客+案件化(リード化)は `crmFacade.intakeFromInquiry` として**実装済みの現行機能**であり②ではない(リデザインでも現行維持。別トラック候補はモック `intakeDone` 等の完了演出強化のみ)。
- **一括移行は非現実的**(現行19 top-levelルート+配下、モック26画面+29モーダル)。Phase 0〜6 の段階移行とする(§4)。
- ドラフト時点の未決3論点(ナビIA・5タブ統合方式・背景色)は冒頭「確定事項」のとおり**すべて確定済み**。実装Issue(R0〜R6c)を起票する。

### 1.2 統合対応(N:1 になる画面)

**(A) 「ホームページ更新」5タブ統合 — works/posts/voices/media/visual → `site`【確定: ルート維持型タブハブ】**

モックは `state.siteTab`(works/posts/voices/media/visual)で1画面に統合。現行は5つの独立ルート+ナビ5項目。**実装方式はルート維持型タブハブに確定**:

| モックタブ | 現行ルート | 実装 |
|---|---|---|
| 施工事例 `works` | `/admin/works`(+`new`/`[id]`) | ルート維持。一覧ページ上部に共通 `SiteSecondaryTabs` を追加 |
| 記事 `posts` | `/admin/posts`(+`new`/`[id]`) | 同上。現行kindタブ(reading/news/blog)はタブ内の二次フィルタとして残す |
| お客様の声 `voices` | `/admin/voices`(+`new`/`[id]`) | 同上 |
| 写真・画像 `media` | `/admin/media` | 同上。「仮画像あり」警告バナーは現行データ(仮素材KPI)で実装可 |
| 見た目の編集 `visual` | `/admin/visual` | 同上。設定画面からの `goSiteBtn`(visualタブ直行)は `/admin/visual` への通常リンクで等価 |

- 実装方式は `src/app/admin/calendar/_ui/secondary-tabs.tsx` の既存パターンを踏襲(`SiteSecondaryTabs` を `src/app/admin/_ui/` に新設し5ページの `page.tsx` 冒頭に配置)。**Next.jsのroute group化(ディレクトリ移動)はしない** — import/テストのchurnが大きく利得がないため。
- ナビは5項目→1項目「ホームページ更新」(href=`/admin/works`)。詳細/編集画面の`PageHeader backHref`は「← ホームページ更新へ」に文言統一。
- URL・ブックマーク・既存テストのルート参照は**すべて無変更**で生き残る(リダイレクト不要)。
- 編集画面(`workEdit`/`postEdit`/`voiceEdit`)は既存 `XxxForm`(mode=create/edit)をそのまま使い、モックの「保存/公開する」2ボタンヘッダに再スタイル。`postEdit`のAI生成注記バナーは post に生成元メタがある場合のみ表示。
- ナビの1項目化タイミング: P1では content 5項目をグループ「①お客さんを作る」配下に暫定残置し(全ルート到達可能性の維持)、P5で `SiteSecondaryTabs` 導入と同時に1項目へ統合して最終14項目とする。

**(B) カレンダー設定ハブ — calendar/{types,templates,connections} → `calSettings`【確定: ルート維持】**

モックは「作業カレンダー(=予定表)」と「カレンダー設定(3タブ: 外部連携/テンプレート/種類)」を分離し、設定へはギアボタン導線。現行は4タブ(予定表含む)の `CalendarSecondaryTabs`。

- `/admin/calendar` = 予定表専用。タブ行を撤去し、ヘッダ右に「⚙ カレンダー設定」ボタン(→ `/admin/calendar/connections`)。
- `/admin/calendar/{connections,templates,types}` の3ページは新 `CalendarSettingsTabs`(3タブ)を共有。`secondary-tabs.tsx` を改修(予定表タブを外し、「← 予定表へ」戻りリンク追加)。
- モックの `calSettings` タブ既定は `connections` なのでギア導線の飛び先も connections とする。
- 設定画面(`/admin/settings`)の「連携の管理」リンク(`goCalSettingsBtn`相当)は `/admin/calendar/connections` へのリンクで等価。

### 3.3 採用判断の経緯(記録)

検討した3案: A=フェーズ別+5タブ統合(URL維持)/ B=グルーピングのみ変更(18項目維持)/ C=現行IA維持・ビジュアルのみ刷新。**2026-07-16 に案Aで確定**。理由:
1. 実ユーザーが単独/少人数の現場運営でありフェーズ型IA(「今どの仕事の段階か」から入る)の効能が最大化される。
2. ルート維持型なら技術コスト・回帰リスクがC(最安全案)と大差ない — `nav-items.ts` のグループ再編+ラベル変更のみで、URLは全て不変、リダイレクト不要。
3. 5タブ統合は「ナビ項目の統合」であって「ページの統合」ではないため、既存のServer Component/フィルタ/ページネーション/テストが無傷。

留意済みリスクと対策:
- 慣れの破壊(問い合わせ・通話の所属グループ移動)→ フェーズ番号表示で新導線を明示。
- コンテンツ編集が1クリック深くなる → タブハブは一覧ページ先頭に常設され、タブ間はワンクリック。
- バッジ件数は毎リクエストのcount取得が必要 → 初期リリースから外し、R6cで縮退設計付きで導入。
- `admin-nav.tsx` の折りたたみ永続化キーはグループid変更に伴い `kumabe-admin-nav-collapsed:v1` → `:v2` へバンプ。
- ナビラベル/グループを参照するテスト・E2Eの文言更新が必要。

## 6. 規約適合性

- **モジュール境界**: 本移行で facade シグネチャ・modules配下は一切変更しない。唯一の例外は R6c のナビバッジ集計で、新規facadeメソッド追加時は既存のエラーコード帯規約に従い新帯を割り当て、`Result<T>` を返す。
- **サーバアクション**: モーダル/シート化はすべて**既存action の呼び出し口の変更**であり、action本体・バリデーション・戻り値契約は不変。
- **テスト**: URL不変方針によりルーティング系テストは無傷。文言(ナビラベル・PageHeaderタイトル)参照テストのみ各Issueで更新箇所を列挙する。`use-escape-to-list` / `use-save-shortcut` / kanbanキーボード / listboxグリッドのアクセシビリティ挙動は「維持」を各Issueの受入基準に明記。
- **モック→実装の翻訳原則**: インラインstyle値はトークン(§2)経由でのみ使用し、任意値クラス(`text-[13.5px]`等)の散布はカレンダーグリッド極小文字などトークン化しない例外に限定する。
